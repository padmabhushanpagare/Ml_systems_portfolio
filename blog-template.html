Designing Feature Stores for Real-time Inference

A deep dive into reducing training-serving skew using Feast and Redis in high-throughput environments.

Machine learning models are only as good as the data they are fed. In production, ensuring that the features served at inference time match those used during training is a critical challenge known as training-serving skew.

The Problem with Batch Pipelines

Traditionally, data teams build batch pipelines that compute features once a day. While this works for analytical dashboards, it fails for real-time fraud detection or recommendation systems where user behavior changes by the second.

Key Requirements

  • Low Latency: Inference feature retrieval must be under 10ms.
  • Consistency: Point-in-time correctness for generating training datasets.
  • Scalability: Handling thousands of requests per second (QPS).

Enter the Feature Store

We implemented Feast (Feature Store) to bridge this gap. By using Redis for online serving and BigQuery for offline storage, we achieved a unified interface for data access. 

# Example Feast definition
from feast import Entity, Feature, FeatureView, ValueType

driver = Entity(name="driver_id", value_type=ValueType.INT64)

driver_stats_view = FeatureView(
    name="driver_stats",
    entities=["driver_id"],
    ttl=timedelta(seconds=86400 * 1),
    features=[
        Feature(name="conv_rate", dtype=ValueType.FLOAT),
        Feature(name="acc_rate", dtype=ValueType.FLOAT),
    ],
    online=True,
    batch_source=driver_stats_source,
    tags={"team": "recommendations"},
)

Results & Impact

This architecture reduced feature drift by 90% and allowed data scientists to push new features to production in hours instead of weeks. The system now handles 5000 QPS with p99 latency < 8ms.
