● Executive Summary

The legacy rule-based system for estimating delivery times resulted in a 25% average error rate, causing significant customer frustration and high support ticket volume. We engineered a regression model integrating real-time traffic, weather, and historical driver performance data. The new system reduced ETA error by 45% and slashed "Where is my order?" tickets by 40% within the first month of deployment.

Business Problem

Customers expect accurate delivery estimates. When an order arrives 15 minutes later than promised, trust erodes. Conversely, over-estimating times leads to cart abandonment. The goal was to minimize the Mean Absolute Error (MAE) of our predictions while maintaining sub-50ms inference latency to avoid slowing down the checkout flow.

Data Overview

We aggregated 6 months of historical order data (~5M records). Key data challenges included high cardinality in pickup locations and noisy GPS signals from driver devices.

Static Features

  • • Pickup/Dropoff Geohashes
  • • Restaurant Preparation Avg
  • • Order Value & Item Count

Dynamic Signals

  • • Real-time Traffic Density
  • • Weather Precipitation
  • • Active Driver Supply

Feature Engineering Strategy

Raw data wasn't enough. We engineered specific features to capture temporal and spatial nuance:

  • Rush Hour Flag: Boolean features for 8-10am and 5-7pm windows based on local time.
  • Cluster Similarity: K-Means clustering on GPS coordinates to group neighborhoods with similar traffic patterns.
  • Driver Stress Index: Moving average of a driver's recent delivery times vs. expected times.

Modeling Approach

We benchmarked Linear Regression, Random Forest, and XGBoost. XGBoost was selected for its handling of non-linear relationships and superior performance on tabular data.

We used 5-fold time-series cross-validation to prevent data leakage (training on future data). Hyperparameters (learning rate, max depth) were tuned using Optuna.

Model Interpretability

Trust is key. We utilized SHAP (SHapley Additive exPlanations) values to understand feature importance. Surprisingly, "Restaurant Prep Time Variance" was often more impactful than "Traffic Density" for short-distance orders.

Production Considerations

Deploying this required a robust MLOps strategy:

  • Latency: Feature store (Redis) lookup takes 4ms, Model Inference (ONNX runtime) takes 15ms. Total P99 latency is 42ms.
  • Monitoring: We set up alerts for Concept Drift. If the distribution of "Traffic Density" inputs shifts significantly (e.g., road closures), the system falls back to a simpler heuristic model.
  • Retraining: Automated pipelines retrain the model weekly on the latest 30 days of data to adapt to seasonality.
